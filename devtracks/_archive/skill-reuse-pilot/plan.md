# Plan: Skill-Wiederverwendungs-Pilot

Siehe [spec.md](spec.md). Status: **geplant / nicht begonnen.**  
**Parent / Exit-Rolle:** [`typst-toolset` Phase 2](../typst-toolset/phase2-skills-starter.md).

Lifecycle: gefundene Skill-Lücken sind der Haupt-Harvest (→ `typst-writer`/`typst-extension` /
package-skill); danach Track nach `_archive/` (siehe `docs/KNOWLEDGE-MAP.md`). Typischerweise
gemeinsam mit Abschluss von `typst-toolset` Phase 2.

## Phasen

- [ ] **Phase 0 — Thema & Form wählen.** Eine bewusst andere Domäne festlegen (z. B. technisches
  Datenblatt oder wissenschaftliche Kurzmitteilung), Umfang ~6–8 Seiten, Zielort `pilots/<name>/`.
  Kann **parallel** zu `typst-toolset` Phase 0–1 laufen.
- [ ] **Gate — Foundation bereit.** Warten auf `typst-toolset` Phase 1 (Package + Smoke), sofern der
  Pilot als Produkt-Reuse zählen soll. Pre-package-Spike nur als optionaler Baseline-Zweig.
- [ ] **Phase 1 — Skill-geführt bauen.** NUR über Skills/Guides + **Foundation-Package** (+ Starter-
  Muster); eigene Token-Palette. Jede genutzte Skill-Regel notieren.
- [ ] **Phase 2 — Reuse-Report.** Protokoll: getriggerte Skills, 1:1-Wiederverwendung vs. Improvisation,
  konkrete Skill-Lücken, grober Aufwand (Zeit/Iterationen); Package-API vs. Improvisation trennen.
- [ ] **Phase 3 — Rückfluss.** Mindestens eine gefundene Lücke als Skill-Verbesserung umsetzen
  (Harvest), Report finalisieren; an `typst-toolset` Phase 2 zurückmelden.

## Abnahme
- Pilot-Artefakt baut reproduzierbar; Reuse-Report + mindestens eine Skill-Verbesserung (oder begründetes „keine Lücken"); post-package-Pfad ohne Showcase-Clone.
